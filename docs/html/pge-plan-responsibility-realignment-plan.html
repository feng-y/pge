PGE Plan Responsibility Realignment 方案评估

该方案是 research.v3 改进之后的下游延续：Research 已经负责把“要解决什么问题”稳定下来，Plan 需要从旧的半发现/半门禁式计划，转为“可执行方案设计 + 可验证执行契约”的阶段。

评估依据：docs/exec-plans/pge-plan-responsibility-realignment-plan.md、当前 pge-research / pge-plan / pge-exec / pge-review 契约、Plan gate 与模板。

结论

总体判断：方向正确，且是 Research 改进的必要延续

这个方案的核心价值不是“重写 Plan 文档”，而是重新划清 Research → Plan → Exec 的责任边界：Research 稳定问题契约，Plan 选择实现路径并产出执行契约，Exec 只执行已通过 plan_gate 的 canonical plan。

解决的主问题旧 pge-plan 仍带有重新发现问题、旧 Research 兼容、重门禁流程和模板负担，导致计划阶段容易膨胀，并把 Exec 真正需要的执行路径、验证耦合和证据要求埋掉。
方案的主优势把 Plan 明确定位为“AI-stable execution path”的设计者：选方法、切 issue、定顺序、定验证拓扑、控制 blast radius，并用 plan_gate 作为唯一执行授权。
可执行性判断如果按方案同步更新 skill、reference、template、eval、review/docs，并做协议一致性审查，它可以把 goal 转成可执行目标；风险在于不要把 Plan Engineering Review 又做成新的重模板。

1. 方案的价值和优势：解决什么问题

1.1 直接解决 Research 更新后的下游错位

当前 pge-research 已经定义了 research.v3：它负责 goal、success shape、scope、non-goals、constraints、context、direction、open questions 和 route。它还明确禁止选择最终实现方案、创建 issue、定义最终 acceptance 或 verification。

因此，如果 pge-plan 继续保留旧逻辑中的“重新确认 intent / 兼容旧 research.v2 / 多个中间 gate 各自产生 route / 固定 DOT 流程”，就会和新的 Research 职责重叠。该方案把 Plan 改为消费 READY_FOR_PLAN 的问题契约，并在其内部选择实现路径，正好补上 Research 改进后的下游责任。

1.2 把 Plan 从“门禁堆叠”转成“执行路径设计”

旧 Plan 能力里有很多有用部件：Coverage Audit、Engineering Review Gate、Inconsistency Grill、Self-Review、Final Plan Gate。但它们容易形成一串 gate ceremony。该方案保留价值，但改变控制权：

  • 保留硬授权：source_contract_check 判断输入是否可计划，plan_gate 判断能否执行。
  • 软化中间机制：Plan Engineering Review、Coverage Audit、Experience Context、Inconsistency Grill、Self-Review 都优先用于修复/硬化 plan，而不是各自制造独立路线。
  • 降低模板负担：LIGHT plan 不应被迫产出深度 gate 表、10/10 评分、固定状态机或无关体验上下文。

1.3 降低 Exec friction

pge-exec 需要的是可派发的 issue contract，而不是一份抽象方案。该计划强化了 Exec 需要的关键字段：behavior delta、target/forbidden areas、acceptance、verification hint、verification coupling、required evidence、dependencies、AFK/HITL、risks 和 terminal conditions。

这能减少三类常见执行失败：

  • Exec 不知道具体改哪些边界，开始发明 scope。
  • issue 表面独立，但验证共享，导致并行执行污染同一工作树。
  • 验收标准来自实现猜测，而不是 Research success shape 或用户目标。

1.4 强化协议型仓库最需要的 coherence

PGE 是技能契约、模板、handoff、eval、route/status/verdict 组成的协议仓库。方案要求任何 protocol/schema/route/API 变化都检查 producer、consumer、validator、reviewer、template、docs、evals 的一致性。这和项目 resident rule 完全同向。

3. 是否能完成 goal 到可以执行的目标

判断：可以，但前提是严格执行方案中的协议一致性更新。

从 goal 到可执行目标的链路应变成：

  1. Research 输出问题契约：goal、success_shape、scope、non_goals、constraints、context、open questions，且 route: READY_FOR_PLAN。
  2. Plan 继承不可静默覆盖的字段：goal/success_shape/scope/non_goals/constraints、Implementation Friction 的 required_plan_adjustment、Progressive Feasibility 的 first_plannable_objective。
  3. Plan 选择实现路径：可以拒绝 Research 的 simplest_direction，但必须说明为什么，并不能改变问题契约。
  4. Plan 切成执行 graph：每个 issue 有 behavior delta、target/forbidden area、acceptance、verification、evidence、dependency、coupling、AFK/HITL、risk 和 state。
  5. Plan Gate 授权执行：只有 plan_gate Verdict: PASS 且 Exec Allowed: yes 才能输出 READY_FOR_EXECUTE。
  6. Exec 消费执行契约：pge-exec 不重新解释外部 plan，不修改 plan，只按 canonical plan.md 执行并产证据。

能完成的原因

  • Plan 的“目标转换”不再是把用户目标摘要成任务列表，而是把 success shape 转成 acceptance / verification / evidence trace。
  • 方案要求明确 first trustworthy verification point，能处理“单 issue 无法独立验证”的情况。
  • 方案把 verification coupling / parallel safety 作为必需执行安全字段，减少计划和执行之间的断裂。
  • Final Plan Gate 的五层检查仍覆盖 contract completeness、engineering review、repo reality、execution readiness、skill execution stability。

仍需注意的风险

  • 命名迁移风险：如果把 Engineering Review Gate 全面改名，但 plan-gate.md、template 或 review 仍只找旧 heading，会造成 validator/consumer 漂移。
  • 过度软化风险：Plan Engineering Review 可以不再是硬授权，但其结果必须被 plan_gate 验证“已消费进 selected approach/issues/acceptance/verification/risks”。
  • Research legacy 删除风险：方案要求不保留旧 Research active compatibility，因此必须同步 eval/docs/references，避免旧字段仍被当作当前 contract。

4. gstack plan-eng-review 覆盖程度

覆盖程度：中高。方案没有把 gstack review 原样搬成独立 gate，而是把它吸收到 Plan Engineering Review，作为计划质量硬化机制。

gstack plan-eng-review 关注点本方案覆盖方式覆盖评价

降低复杂度明确禁止 speculative helpers/abstractions/flags/frameworks；要求 blast-radius minimization 和 complexity/risk reduction。覆盖。

5. 建议的实施重点

  1. 先改 skills/pge-plan/SKILL.md 的职责定位：把 Plan 的核心职责改为 executable solution design，删除或降级旧 Research compatibility 文案。
  2. 再改 Plan Engineering Review 参考：把当前 engineering-review-gate.md 从“独立 hard gate”改为 Plan-owned decision-hardening pass，并保留 depth scaling。
  3. 同步 plan-gate.md：它仍是唯一执行授权 validator；检查 Plan Engineering Review 是否已被消费，而不是依赖旧多 gate 流程。
  4. 收轻模板：保留 exec-required 字段，移除默认 10/10/rating/深 gate ceremony；强化 behavior delta、verification coupling、evidence、forbidden areas、HITL。
  5. 补 eval：覆盖 Research override protection、simplest_direction 可被拒绝、blocking question 不变 assumption、Progressive Feasibility、Implementation Friction、LIGHT compact、protocol coherence。
  6. 做协议一致性审查：按 producer/consumer/validator/state writer/final response 逐项核对 README、CLAUDE、skills、references、templates、evals、review、exec。

最终评价

该方案是 Research 改进后的正确下一步：它让 Research 停在“问题契约”，让 Plan 真正承担“实现路径与执行契约设计”，让 Exec 不再补计划阶段的空洞。

它与旧 Plan 的关键能力基本对齐：保留 plan.v2、canonical plan.md、source readiness、approach selection、Architecture Delta Contract、Fast Lane/Fast Adopt、Final Plan Gate 和 Exec 必需字段；同时删除/软化那些造成计划膨胀的旧门禁仪式。

如果实施时把命名、模板、eval、review/exec 消费关系同步好，它能够把 goal 转成可执行目标，并且对 gstack plan-eng-review 的覆盖达到“计划质量硬化”所需的程度。最大注意点是：不要把 Plan Engineering Review 变成新的重模板，也不要继续把旧 Research 字段当作 active contract。
