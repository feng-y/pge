# I004: Align Evaluator And Review Boundaries

Intent: prevent the new issue files from becoming final proof or a checklist-style review target.

## introduction

Align Evaluator and review wording so issue files remain supporting execution evidence while final judgment stays at the run/plan outcome level.

## evaluation

This is boundary hardening. It prevents the new issue-file contract from turning Evaluator or review into per-issue checklist approval.

## task

Align Evaluator, evaluator threshold, and review wording so issue files support execution and evidence traceability without becoming a final acceptance checklist.

## behavior_contract

- Current Behavior: some wording may imply issue completion or issue checklist review is primary proof.
- Desired Behavior: Evaluator/review judge the composed diff and evidence against `plan.md`, original intent, acceptance, verification, and non-goals.
- Behavior Delta: clarify issue files as supporting contracts/evidence while final pass/fail remains run-level and plan-outcome based.
- Key Interfaces: `skills/pge-exec/handoffs/evaluator.md`, `skills/pge-exec/references/evaluator-thresholds.md`, `skills/pge-review/SKILL.md`.
- Trigger Predicate: evaluator/review runs after issue-file-based execution.
- Output Admission Predicate: evaluation can fail a run even if individual issues report completion when plan-level outcome is unmet.
- Out Of Scope Confirmed: do not remove bounded issue repair through `pge-exec`.
- What Not To Infer: do not approve ship from issue completion alone.

## scope

### Do

- Clarify issue files as supporting evidence.
- Preserve run-level plan outcome evaluation.
- Preserve bounded in-contract repair semantics.

### Do Not

- Do not add per-issue Evaluator approvals as required gates.
- Do not change review route vocabulary.
- Do not create a new final proof artifact.
