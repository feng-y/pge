Claude Code 大型代码库最佳实践

来源：Anthropic Blog, 2026-05-14。 核心结论：模型周围的 harness（CLAUDE.md、hooks、skills、plugins、MCP）决定 Claude Code 在大型代码库中的表现，比模型本身更重要。

核心洞察：Claude Code 像工程师一样导航代码库（遍历文件系统、grep、跟踪引用），不需要 RAG 索引。但它需要足够的起始上下文知道从哪里找。Harness 的质量决定导航质量。

Harness > Model

最常见的误解：Claude Code 的能力完全由模型决定。实际上，围绕模型构建的生态系统（harness）决定了实际表现。Harness 由五个扩展点构成，构建顺序很重要——每一层建立在前一层之上。

五个扩展点详解

CLAUDE.md — 基础上下文层

  • 每次会话自动加载：root 文件给全局视图，子目录文件给局部约定
  • 保持精简：只放广泛适用的内容，膨胀会拖累性能
  • 分层加载：Claude 沿目录树向上走，逐层加载每个 CLAUDE.md

常见错误：把应该放在 skill 里的可复用专业知识塞进 CLAUDE.md

Hooks — 自我改进层

  • Stop hook：会话结束时反思发生了什么，提议 CLAUDE.md 更新
  • Start hook：动态加载团队特定上下文，每个开发者自动获得正确配置
  • 对 lint/format 等自动检查，hooks 比依赖 Claude 记住指令更一致

常见错误：用 prompt 做应该自动运行的事情

Skills — 按需专业知识

  • 渐进式披露：专业工作流和领域知识只在任务需要时加载
  • 可以绑定到特定路径（如支付服务的部署 skill 只在该目录激活）
  • 避免所有专业知识竞争上下文空间

常见错误：把所有东西都塞进 CLAUDE.md 而不是用 skill

Plugins — 分发层

  • 将 skills、hooks、MCP 配置打包为单个可安装包
  • 新工程师第一天安装即获得与老手相同的上下文和能力
  • 通过 managed marketplace 全组织分发更新

常见错误：让好的配置停留在部落知识阶段

MCP Servers — 外部连接

  • 连接 Claude 到内部工具、数据源、API
  • 最成熟的团队：构建 MCP server 暴露结构化搜索作为 Claude 可直接调用的工具
  • 其他用途：内部文档、工单系统、分析平台

常见错误：在基础还没搞好时就去建 MCP 连接

补充能力：LSP 与 Subagents
LSP 集成

给 Claude 与 IDE 相同的符号级导航。没有 LSP 时，Claude 用文本模式匹配，可能落在错误的符号上。对多语言代码库（尤其 C/C++）是最高价值投资之一。

一家企业软件公司在 Claude Code 部署前全组织部署了 LSP 集成，专门为了让 C/C++ 导航在规模上可靠。

Subagents

隔离的 Claude 实例，有自己的上下文窗口。模式：启动只读 subagent 映射子系统并写入文件，然后主 agent 带着全貌进行编辑。将探索与编辑分离。

三个配置模式

模式 1

让代码库在规模上可导航

  • 精简分层 CLAUDE.md：root 文件只放指针和关键陷阱，其他都是噪音
  • 在子目录初始化，不在 repo root：Claude 自动向上走加载所有 CLAUDE.md
  • 按子目录限定 test/lint 命令：改一个服务时跑全套会超时并浪费上下文
  • 用 .ignore 排除生成文件：在 .claude/settings.json 提交 permissions.deny 规则
  • 构建代码库地图：当目录结构不能自解释时，在 repo root 放一个轻量 markdown 列出顶层文件夹
  • 运行 LSP：让 Claude 按符号搜索而非字符串搜索
模式 2

随模型演进主动维护 CLAUDE.md

  • 为当前模型写的指令可能对未来模型起反作用
  • 例：告诉 Claude 把每次重构拆成单文件变更 → 阻止新模型做协调的跨文件编辑
  • 补偿特定模型限制的 skills/hooks 在限制消失后变成开销
  • 节奏：每 3-6 个月做一次有意义的配置审查，加上每次重大模型发布后
模式 3

指定 Claude Code 管理和采纳的所有权

  • 最快扩散的部署：广泛访问前有专门的基础设施投入
  • 一个小团队（甚至一个人）在开发者首次接触前就接好了工具
  • 组织归属：Developer Experience / Developer Productivity 团队
  • 新兴角色：Agent Manager — 混合 PM/工程师，专门管理 Claude Code 生态
  • 最小可行版本：一个 DRI，拥有配置、权限策略、plugin marketplace、CLAUDE.md 约定的所有权
  • 没有集中化 → 自下而上的采纳产生热情但碎片化，知识停留在部落阶段

组织治理（受监管行业）

  • 从定义好的已批准 skills 集合开始
  • 要求代码审查流程（AI 生成代码与人类代码相同标准）
  • 有限的初始访问，随信心增长扩展
  • 尽早建立跨职能工作组（工程 + 信息安全 + 治理）

对 PGE 的启示

PGE 已经做对的
  • Skills 作为按需专业知识（pge-research、pge-plan、pge-exec 等）
  • Plugin 分发（.claude-plugin/plugin.json + marketplace.json）
  • CLAUDE.md 分层（repo root + skill 级别）
  • Subagent 模式（Generator/Evaluator 作为隔离 Agent Teams）
可以改进的方向
  • Hooks 自我改进：考虑 stop hook 在 pge-exec 完成后提议 CLAUDE.md 更新
  • LSP 集成：当前 pge-exec Generator 依赖 grep/read，LSP 可以提升符号导航精度
  • 配置审查节奏：随模型升级定期审查 skill 约束是否仍然必要
  • 代码库地图：README.md 已经做了这个工作，但可以更结构化
